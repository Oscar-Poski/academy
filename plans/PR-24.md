# PR-24 Plan: Backend Completion Gating Enforcement (`POST /v1/progress/sections/:sectionId/complete`)

## Summary
PR-24 enforces server-side gating on section completion so users cannot complete gated sections by bypassing UI state.

Goals:
- Add completion gating to existing `POST /v1/progress/sections/:sectionId/complete`.
- Enforce quiz-pass requirements for quiz sections.
- Enforce unlock requirements in a completion-safe way (no prerequisite deadlock on the section being completed).
- Return structured `409` gating responses while keeping existing success/idempotency behavior.

Out of scope:
- No schema/migrations.
- No web/UI changes in this PR.
- No credits redemption/min-level enforcement.
- No changes to unlock endpoints themselves.

## Important Public API / Interface Changes

## Endpoint behavior change
Existing endpoint:
- `POST /v1/progress/sections/:sectionId/complete`

New behavior:
- returns `409 Conflict` with structured payload when gating requirements are unmet.

### New error payload DTO
Create `/Users/poski/academy/apps/api/src/modules/progress/dto/complete-section-gating-error.dto.ts`:

```ts
export interface CompleteSectionGatingErrorDto {
  code: 'completion_blocked';
  reasons: string[];
  requiresQuizPass: boolean;
  requiresUnlock: boolean;
}
```

### Status codes for `complete`
- `201` success (unchanged)
- `400` missing/unknown user (unchanged)
- `404` section not found (unchanged)
- `409` gating blocked (new)

## Gating Rules (Decision Complete)

## 1) Quiz gate
For target section:
- If `section.hasQuiz === true`, require latest quiz attempt for `(userId, sectionId)` to exist and `passed === true`.
- Latest ordering: `submittedAt DESC`, `attemptNo DESC`, `id ASC` (match existing unlock/quiz semantics).
- If unmet, add reason:
  - `Pass quiz before completing this section.`

## 2) Unlock-aware gate
Evaluate module unlock via existing unlock engine:
- Use `UnlocksService.getModuleStatusForKnownUser(userId, moduleId)`.

If unlock decision is locked:
- start from `unlockDecision.reasons`.
- remove only the self-prerequisite reason that would deadlock completion:
  - exact string: `Complete prerequisite section: <currentSectionId>`
- remaining reasons (if any) block completion.

Reason string passthrough policy:
- include remaining unlock reasons directly in gating error payload.
- dedupe final reason list.

### Why self-prerequisite filtering is required
Seeded module prerequisite rules can include the section being completed; without this filter, completion becomes impossible.

## 3) Final block condition
If any reasons remain after quiz + unlock checks:
- throw `ConflictException` with `CompleteSectionGatingErrorDto`.

---

## Idempotency / Mutation Semantics

1. Preserve existing idempotency:
- If section is already completed for user, return existing completed progress immediately (no gating re-check required).

2. Gating happens before mutation for non-completed rows:
- avoid creating/updating completion when blocked.

3. Continue existing completion flow after gate passes:
- ensure progress row exists (`startSection` behavior retained)
- update status to completed with `completionPct=100`, timestamps as today.

Add dependencies/helpers:
- Inject `UnlocksService`.
- helper: `assertCanCompleteSection(userId: string, sectionId: string): Promise<void>`
- helper: `getLatestQuizPassState(userId, sectionId): Promise<boolean>`
- helper: `buildCompletionBlockedError(reasons, flags): ConflictException`

Flow in `completeSection`:
1. `assertKnownUser`.
2. Read existing progress row by `(userId, sectionId)`.
3. If existing and `status === completed`, return as-is.
4. Run `assertCanCompleteSection(...)`.
5. Continue existing completion logic (start/complete).

`assertCanCompleteSection`:
1. Load section `{ id, moduleId, hasQuiz }` or `404`.
2. Build `reasons[]`, flags:
   - `requiresQuizPass` if quiz unmet.
   - `requiresUnlock` if unlock reasons remain after self-prereq filtering.
3. If reasons not empty, throw `ConflictException({ code: 'completion_blocked', ... })`.

## Explicit Assumptions and Defaults
- Quiz gating applies when `sections.hasQuiz === true`.
- Quiz pass decision is based on **latest** attempt only.
- Unlock gating reuses current unlock reason semantics from PR-21.
- Self-prerequisite unlock reason for the section being completed is ignored for completion checks to avoid deadlock.
- Credits/min-level rule enforcement remains deferred; unsupported unlock rules continue to surface as reasons and can block completion if active.
