# PR-14 Plan: Admin Content Import Endpoint (`POST /v1/admin/content/import`) Using `content-importer`

## Summary

PR-14 adds the first admin/content operations API endpoint in `apps/api`:

- `POST /v1/admin/content/import`
- accepts a **server-local bundle path** and `mode` (`dryRun` / `apply`)
- invokes the existing `/Users/poski/academy/packages/content-importer` package from PR-12/PR-13
- returns the importer’s structured report (parse + apply results)

This PR is API-only and intentionally compact:
- no auth/access control yet (deferred)
- no publish/version management endpoints yet (PR-15)
- no file upload support (local path only)

## Current State (Repo-Grounded)

- `packages/content-importer` already exposes:
  - `parseContentBundle(...)`
  - `createDryRunApplyReport(...)`
  - `applyParsedContentReport(...)`
  - `ContentImportApplyReport` types
- `apps/api` currently has modules:
  - `content`, `progress`, `analytics`
- No `admin` module exists yet
- `apps/api` does not use global validation pipes, so request validation is currently service/manual style
- `PrismaService` extends `PrismaClient`, which can be passed to importer apply helpers to reuse the API DB connection

## Scope (Locked In)

## Important Public API / Interface Additions

### New Endpoint
- `POST /v1/admin/content/import`

### Request Body (snake_case + mode string)
```ts
interface ImportContentRequestDto {
  bundle_path: string;
  mode: 'dryRun' | 'apply';
}
```

### Response Body
Return the importer’s structured report directly (PR-13 type shape):
- `ContentImportApplyReport`

Behavior by mode:
- `dryRun` => `mode: "dry_run"`, `applied: false`, zero apply counts
- `apply` => `mode: "apply"`, `applied: true|false` depending on parse error abort

### HTTP status codes (locked)
- `200 OK` on successful dry-run/apply execution (including parse-error apply abort report)
- `400 Bad Request` for:
  - missing/invalid `bundle_path`
  - missing/invalid `mode`
  - unreadable/nonexistent bundle path
  - path outside `CONTENT_IMPORT_ROOT` (if env guard enabled and configured)
- `500` for unexpected importer/apply runtime errors

## Controller Plan (`admin.controller.ts`)

### Route prefix
- `@Controller('v1/admin')`

### Endpoint
- `@Post('content/import')`

Method signature:
- `importContent(@Body() body: ImportContentRequestDto): Promise<ImportContentResponseDto>`

Controller is thin:
- delegates to `AdminService.importContent(body)`

## Service Plan (`admin.service.ts`)

### Dependencies
- Inject `PrismaService`
- Import from `@academy/content-importer`:
  - `parseContentBundle`
  - `createDryRunApplyReport`
  - `applyParsedContentReport`
  - type `ContentImportApplyReport`

### Main method
- `async importContent(body: ImportContentRequestDto): Promise<ContentImportApplyReport>`

### Request validation (manual, locked)
Validate in service using `BadRequestException`:
1. `bundle_path`
   - required string
   - trim + non-empty
2. `mode`
   - required string
   - must be exactly `'dryRun'` or `'apply'`

No class-validator in PR-14.

### Path handling and safety (locked)
1. Resolve `bundle_path` to absolute path (`path.resolve(...)`)
2. Optional env guard:
   - if `process.env.CONTENT_IMPORT_ROOT` is set and non-empty:
     - resolve it to absolute path
     - require `bundle_path` to be within that root
     - else `400`
3. Invoke importer parse/apply using resolved path

Reason:
- Keeps endpoint useful for local dev immediately
- Adds a minimal safety control without introducing ConfigModule/auth
- Compatible with roadmap mention of `CONTENT_IMPORT_ROOT`

### Mode execution
- `dryRun`
  - `const parseReport = await parseContentBundle(resolvedPath)`
  - return `createDryRunApplyReport(parseReport)`
- `apply`
  - `const parseReport = await parseContentBundle(resolvedPath)`
  - return `applyParsedContentReport(parseReport, { prisma: this.prisma })`
  - Reuses API Prisma connection; no extra Prisma client instance

### Error mapping (locked)
Catch importer/fs path errors and map to `400`:
- `ENOENT`, `ENOTDIR`, `EACCES`, `EPERM`
- message example: `bundle_path is not readable: <resolvedPath>`

Let unexpected errors bubble (500), especially DB write failures in `apply` mode.

### Parse-error apply behavior (important)
If importer returns:
- `mode: "apply"`
- `applied: false`
- `abortedReason: "parse_errors"`

This is still returned as `200`, not a thrown `400`.
Reason:
- It is a valid import report and consistent with CLI semantics.

## E2E Test Plan (`apps/api/test/admin-content-import.e2e-spec.ts`)

Create a dedicated e2e suite (do not overload existing analytics/content/progress tests).

### Test setup pattern
Reuse existing e2e pattern:
- `Test.createTestingModule({ imports: [AppModule] })`
- `INestApplication`
- `PrismaClient` for assertions/cleanup

### Test data strategy (locked)
Use **temporary bundle directories/files with unique slugs** instead of the shared fixture bundle to avoid mutating seeded rows (`request-response-cycle v2`, etc.) and causing cross-test interference.

Generate temp markdown files during test with:
- unique `path_slug/module_slug/section_slug`
- one version (e.g., `version_number: 1`)
- valid frontmatter + markdown body

Cleanup after each:
- delete created DB rows by unique slugs (sections/modules/paths cascade should handle children)
- remove temp dir

### Required test cases

#### 1. `dryRun` returns parse/apply report and does not write DB
- Create temp bundle with unique slugs
- `POST /v1/admin/content/import` `{ bundle_path, mode: 'dryRun' }`
- Expect `200`
- Assert response:
  - `mode === 'dry_run'`
  - `applied === false`
  - `parseReport.errorCount === 0`
- Assert DB:
  - no `path` row exists with temp `path_slug`

#### 2. `apply` writes draft content and returns apply report
- Use temp bundle with unique slugs
- `POST ... { bundle_path, mode: 'apply' }`
- Expect `200`
- Assert response:
  - `mode === 'apply'`
  - `applied === true`
  - counts show created rows (`pathsCreated`, `modulesCreated`, `sectionsCreated`, `sectionVersionsCreated`)
- Assert DB:
  - created `path/module/section` rows exist
  - `sectionVersion.status === draft`
  - `lessonBlock` row exists with `blockType === markdown`

#### 3. repeated `apply` is idempotent (no duplicate version)
- Call same apply request twice
- Expect `200` both times
- Assert second response:
  - `sectionVersionsCreated === 0`
  - `sectionVersionsUpdated >= 1`
- Assert DB:
  - exactly one `sectionVersion` row for `(sectionId, version_number)`

#### 4. invalid request body rejected
- Missing `bundle_path` or invalid `mode`
- Expect `400`

#### 5. nonexistent bundle path rejected
- `bundle_path` points to missing dir
- Expect `400`

### Optional low-friction test
#### 6. `apply` parse errors return report (200) and do not write
- Temp bundle with malformed frontmatter
- `mode: 'apply'`
- Expect `200`
- Assert:
  - `applied === false`
  - `abortedReason === 'parse_errors'`
  - no DB rows created

(Useful because this behavior is non-obvious and central to PR-14.)

## Acceptance Criteria

- `POST /v1/admin/content/import` exists in `apps/api`.
- Endpoint accepts `{ bundle_path, mode }` and returns importer report JSON.
- `dryRun` mode parses bundle and returns report without DB writes.
- `apply` mode invokes importer DB upsert logic via API Prisma connection.
- Parse-error apply requests return a non-writing report (`applied: false`, `abortedReason: "parse_errors"`).
- Invalid request body and invalid bundle paths return `400`.
- e2e coverage exists for dryRun/apply and invalid request/path cases.
- No Prisma schema/migration changes.
- No web changes.

## Explicit Assumptions and Defaults

- No auth is enforced on admin endpoints yet (consistent with current MVP stage); this is temporary.
- `bundle_path` is a server-local filesystem path (not upload).
- `mode` values are exactly `'dryRun'` and `'apply'` (camel-case string literals in request body).
- The endpoint returns importer report JSON with `200` even when `apply` aborts due to parse errors.
- `CONTENT_IMPORT_ROOT` guard is optional: enforced only when the env var is set.
- PR-14 reuses PR-13 importer semantics exactly (including non-draft version skips and idempotent draft updates).
