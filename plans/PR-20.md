## Public API Additions

## New endpoint
- `GET /v1/unlocks/modules/:moduleId/status`

Headers:
- Required: `x-user-id`

Response:
```ts
interface UnlockDecisionDto {
  moduleId: string;
  isUnlocked: boolean;
  reasons: string[];
  requiresCredits: boolean;
  creditsCost: number;
}
```

Status codes:
- `200` success
- `400` missing/unknown `x-user-id`
- `404` module not found
- `500` malformed active rule config (server-side content data issue)

---

## Unlock Evaluation Rules (PR-20 locked behavior)

## 1) Base module metadata
- Load module by `moduleId`.
- Use module `creditsCost` to populate response:
  - `requiresCredits = module.creditsCost > 0`
  - `creditsCost = module.creditsCost`

PR-20 does **not** enforce credits; fields are informational.

## 2) Rule scope filter
- Evaluate only active rules:
  - `scopeType = module`
  - `scopeId = moduleId`
  - `isActive = true`
- Order by `priority DESC`, `createdAt ASC`, `id ASC` (deterministic).
- All active rules are ANDed (all must pass for unlock).

## 3) Supported rule type in PR-20
- `prereq_sections` only.

`rule_config_json` contract:
```json
{ "section_ids": ["<sectionId1>", "<sectionId2>"] }
```

Validation:
- `section_ids` must be non-empty array of non-empty strings.
- malformed config => `500` (misconfigured rule data).

Evaluation:
- Required sections pass if user has `user_section_progress.status = completed` for each section ID.
- Missing/incomplete required sections make rule fail.
- For each missing section, add reason:
  - `Complete prerequisite section: <sectionId>`

## 4) Unsupported types in PR-20
- `quiz_pass`, `credits`, `min_level` rules are treated as not satisfied with explicit reason:
  - `Unsupported unlock rule in PR-20: <ruleType>`
- This keeps behavior safe until PR-21+ implements them.

## 5) No-rule behavior
- If no active rules exist for module scope:
  - `isUnlocked = true`
  - `reasons = []`

## 6) UserUnlock table
- Ignore `user_unlocks` in PR-20 (read-only evaluator only, no persisted grants used yet).
- PR-21 will add evaluate/persist semantics.

## New e2e suite
Create `/Users/poski/academy/apps/api/test/unlock-status.e2e-spec.ts`

Use seeded module/section/rules:
- module slug: `http-basics-module`
- prerequisite section slug: `request-response-cycle`

Create test users similarly to other suites.

### Required test cases

1. **Locked before prerequisite completion**
- User has no completed progress for prerequisite section.
- `GET /v1/unlocks/modules/:moduleId/status` => `200`
- assert:
  - `isUnlocked = false`
  - `reasons` contains prerequisite reason
  - `moduleId` matches

2. **Unlocked after prerequisite completion**
- Mark prerequisite section completed via progress endpoint:
  - `POST /v1/progress/sections/:sectionId/complete` with same user
- Re-check status.
- assert:
  - `isUnlocked = true`
  - `reasons` empty (or no unmet reasons)

3. **Missing/unknown user header**
- missing `x-user-id` => `400`
- unknown `x-user-id` => `400`

4. **Unknown module**
- non-existent moduleId => `404`

5. **No active rules => unlocked**
- create temp module without rules in test setup
- check status => unlocked true, empty reasons
- cleanup temp module after test

### Optional test
6. **Malformed prereq rule config yields 500**
- insert temp malformed active rule for temp module
- verify endpoint returns `500`
- cleanup row/module

## Explicit Assumptions and Defaults
- Rule semantics are AND across active module-scope rules.
- PR-20 officially supports only `prereq_sections`; other rule types return unmet/unsupported reasons.
- Credits fields are informational only in PR-20.
- Missing/malformed rule config is treated as server misconfiguration (`500`), not silently ignored.
- Unlock persistence and explicit evaluate endpoint are deferred to PR-21.
