# PR-22 Plan: Content API Lock Metadata (Additive, User-Aware)

## Summary
PR-22 adds **optional lock metadata** to existing content API payloads when `x-user-id` is supplied and resolvable to a real user.

Goals:
- Extend content payloads with lock state for module/section navigation decisions.
- Keep responses backward-compatible for anonymous callers.
- Reuse PR-21 unlock logic (`prereq_sections`, `quiz_pass`, persisted `user_unlocks`) without changing schema.

Out of scope:
- No new endpoints.
- No schema/migration/seed changes.
- No web changes in PR-22.
- No credits redemption/min-level enforcement.

## Important Public API / Interface Additions

## Existing endpoints (additive behavior)
- `GET /v1/paths/:pathId` now optionally accepts `x-user-id`.
- `GET /v1/modules/:moduleId` now optionally accepts `x-user-id`.
- `GET /v1/sections/:sectionId` already accepts optional `x-user-id`; add lock metadata in navigation.

No URL/path changes and no required headers added.

## New content lock DTO
Create content-local DTO:

```ts
export interface ContentLockMetadataDto {
  isLocked: boolean;
  reasons: string[];
  requiresCredits: boolean;
  creditsCost: number;
}
```

## DTO additions (all optional/additive)
- `PathTreeModuleDto`:
  - `lock?: ContentLockMetadataDto`
- `PathTreeSectionDto`:
  - `lock?: ContentLockMetadataDto` (derived from parent module in PR-22)
- `ModuleDetailDto`:
  - `lock?: ContentLockMetadataDto`
- `ModuleDetailSectionDto`:
  - `lock?: ContentLockMetadataDto` (derived from module in PR-22)
- `SectionNavigationDto`:
  - `prevSectionLock?: ContentLockMetadataDto | null`
  - `nextSectionLock?: ContentLockMetadataDto | null`

### Backward compatibility contract
- If `x-user-id` is absent or not a known user, **new lock fields are omitted**.
- Existing fields and ordering remain unchanged.

## Implementation Plan

## 1) Reuse unlock logic cleanly
Use `UnlocksService` from content layer instead of duplicating policy logic.

### `unlocks.module.ts`
- Export `UnlocksService`:
  - `exports: [UnlocksService]`

### `content.module.ts`
- Import `UnlocksModule` so `ContentService` can inject `UnlocksService`.

### `unlocks.service.ts`
Add an internal reusable method for known users to avoid repeated user validation from content:

- `getModuleStatusForKnownUser(userId: string, moduleId: string): Promise<UnlockDecisionDto>`

Behavior:
- same as `getModuleStatus`, but skips `assertKnownUser`.
- still honors persisted `user_unlocks` + active rules (`prereq_sections`, `quiz_pass`) + unsupported-rule reasons.
- keep existing public `getModuleStatus` and `evaluateModuleUnlock` unchanged.

## 2) Content controller header wiring
### `content.controller.ts`
Update signatures:
- `getPath(@Param('pathId') pathId: string, @Headers('x-user-id') userId?: string)`
- `getModule(@Param('moduleId') moduleId: string, @Headers('x-user-id') userId?: string)`

Pass `userId` through to service methods.

## 3) Content service lock metadata enrichment
### `content.service.ts`
- Inject `UnlocksService`.
- Update methods:
  - `getPathTree(pathId: string, userId?: string)`
  - `getModule(moduleId: string, userId?: string)`
  - `getSection(sectionId: string, userId?: string)` (existing signature retained)

### Add helper: `resolveKnownUserIdForLockContext(userId?: string): Promise<string | null>`
- `null` when header absent/blank.
- `null` when user does not exist.
- return trimmed user ID when user exists.
- never throws for unknown/missing user (preserves content endpoint behavior).

### Add helper: `getModuleLockMetadata(moduleId: string, knownUserId: string): Promise<ContentLockMetadataDto>`
- calls `unlocksService.getModuleStatusForKnownUser(...)`
- maps `UnlockDecisionDto` to `ContentLockMetadataDto` (drops `moduleId`).

### Path tree enrichment (`GET /v1/paths/:pathId`)
- When known user exists:
  - compute lock for each module in returned path.
  - attach `module.lock`.
  - attach same lock to each section in that module (`section.lock`) for PR-22.
- When no known user:
  - return current payload with no lock fields.

### Module detail enrichment (`GET /v1/modules/:moduleId`)
- When known user exists:
  - compute module lock once.
  - set `module.lock`.
  - attach same lock to all `sections[].lock`.
- Else omit lock fields.

### Section detail enrichment (`GET /v1/sections/:sectionId`)
- Keep current version resolution and lesson block behavior unchanged.
- When known user exists:
  - compute module lock for `section.moduleId`.
  - add:
    - `navigation.prevSectionLock = prevSectionId ? moduleLock : null`
    - `navigation.nextSectionLock = nextSectionId ? moduleLock : null`
- Else omit the new navigation lock fields.

## 4) Error handling policy for lock metadata (locked decision)
- If lock metadata evaluation fails due unlock rule misconfiguration or unexpected unlock-service errors:
  - **do not fail content response**.
  - omit lock metadata fields for that response.
- Reason: PR-22 is metadata-only; content retrieval must stay resilient.

2. **Unknown `x-user-id` behaves like anonymous (non-fatal)**

3. **Known locked user gets lock metadata**
- Create test user with no prereq completion/unlock.
- `GET /v1/paths/:pathId` with `x-user-id`:
  - seeded module has `lock.isLocked === true`
  - reasons include `Complete prerequisite section: <sectionId>`
  - sections under module inherit locked metadata.
- `GET /v1/modules/:moduleId` with `x-user-id`:
  - module + sections include locked metadata.
- `GET /v1/sections/:sectionId` with `x-user-id`:
  - if next/prev exists, corresponding nav lock fields present and locked.

4. **Known unlocked user gets unlocked metadata**
- Complete prereq section and call `POST /v1/unlocks/modules/:moduleId/evaluate`.
- Re-fetch content endpoints with same user.
- Assert lock metadata shows `isLocked === false`, `reasons === []`.

5. **No regression in ordering and core payload**
- Assert module/section/block ordering remains same as existing tests.

## Acceptance Criteria
- Content endpoints include lock metadata only when `x-user-id` resolves to a known user.
- `GET /v1/paths/:pathId` and `GET /v1/modules/:moduleId` accept optional `x-user-id`.
- `GET /v1/sections/:sectionId` includes additive navigation lock metadata for known users.
- Existing anonymous behavior and payload compatibility remain intact.
- Lock metadata uses PR-21 unlock semantics (`user_unlocks`, `prereq_sections`, `quiz_pass`).
- No schema/migration/seed changes.
- API typecheck/tests pass, including new content-lock-metadata e2e coverage.

---

## Explicit Assumptions and Defaults
- In PR-22, section-level lock metadata is derived from module-level unlock status (section rules not introduced yet).
- Unknown `x-user-id` on content endpoints is treated as anonymous context, not an error.
- Lock metadata failures are non-fatal for content retrieval in PR-22 (metadata omitted).
- Credits fields remain informational only (`requiresCredits`, `creditsCost`), no redemption logic yet.
