## API Validation Rules (Decision Complete)

### `event_name` allowlist (strict in PR-11)
Accept only:
- `section_start`
- `section_complete`
- `player_exit`
- `player_dropoff`

Reject any other `event_name` with `400`.

### `occurred_at`
- Required
- Must parse to valid timestamp (`Date`)
- `400` on missing/invalid (existing behavior retained)

### `idempotency_key`
- Optional
- If provided, must be a non-empty trimmed string
- Empty/whitespace-only value => treat as invalid (`400`)
- Missing/`null` => no dedupe applied (PR-10 behavior retained)

### `payload_json`
- If omitted: default to `{}`
- If provided: must be a JSON object (not array, not primitive, not null) in PR-11
- `400` if invalid type

Reason:
- This gives meaningful validation without overcommitting to event-specific schemas yet.

## Prisma / DB Changes

### Schema change
Add nullable unique column to `AnalyticsEvent` model:
```prisma
idempotencyKey String? @unique @map("idempotency_key")
```

Updated model excerpt:
```prisma
model AnalyticsEvent {
  id               String   @id @default(cuid())
  idempotencyKey   String?  @unique @map("idempotency_key")
  userId           String?  @map("user_id")
  eventName        String   @map("event_name")
  ...
}
```

### Migration expectations
- `ALTER TABLE "analytics_events" ADD COLUMN "idempotency_key" TEXT;`
- `CREATE UNIQUE INDEX "analytics_events_idempotency_key_key" ON "analytics_events"("idempotency_key");`

Notes:
- Postgres allows multiple `NULL`s in a unique index, so nullable unique is sufficient.
- No backfill required.

## API Service Implementation Plan (`analytics.service.ts`)

### Core changes
1. Add `ALLOWED_EVENT_NAMES` constant (readonly set)
2. Validate `event_name` against allowlist after trimming
3. Validate `payload_json` is object and normalize to `{}` if missing
4. Validate/normalize `idempotency_key`
5. Ingest with dedupe behavior:
   - if no `idempotency_key`: create row (current behavior)
   - if key provided:
     - attempt create
     - on Prisma unique constraint violation (`P2002`) for `idempotency_key`, fetch existing row by key and return its ID

### Dedupe behavior details (locked in)
- Do not compare payload equality on replay in PR-11
- First writer wins
- Replays with same key return the originally persisted event ID

### Helper methods to add
- `validateEventName(value: unknown): string`
- `parseOccurredAt(value: unknown): Date`
- `normalizeIdempotencyKey(value: unknown): string | null`
- `normalizePayloadObject(value: unknown): Record<string, unknown>`

## Web Event Emission Plan

### 1) Emit `section_start` in `/Users/poski/academy/apps/web/app/learn/[sectionId]/page.tsx` (server-side)
Hook point:
- After `startSectionProgress(section.id)` resolves successfully (already available as `sectionProgress`)
- After `getModule(...)` resolves (for `pathId` via `module.pathId`)
- Before render returns (best-effort, errors swallowed)

Emission condition:
- Only emit if `sectionProgress` is non-null

Event payload:
```ts
{
  event_name: 'section_start',
  occurred_at: new Date().toISOString(),
  idempotency_key: `section_start:${sectionProgress.id}`,
  user_id: sectionProgress.userId,
  path_id: module.pathId,
  module_id: module.id,
  section_id: section.id,
  section_version_id: sectionProgress.sectionVersionId,
  payload_json: {
    source: 'learn_page'
  }
}
```

Behavior:
- Wrap in `.catch(() => null)` (or `try/catch`) so analytics failure does not break page render

Why key uses `sectionProgress.id`:
- Stable for the user+section progress row
- Dedupes page refresh/re-renders for the same started section row

- Best-effort (analytics failure should not block UI success path)

- `sectionId`

- `pathId: string`
- `moduleId: string`

These are available in `PlayerContent` via `breadcrumb`.

`PlayerContent` change:

Emission payload:
```ts
{
  event_name: 'section_complete',
  occurred_at: new Date().toISOString(),
  idempotency_key: `section_complete:${result.id}`,
  user_id: result.userId,
  path_id: pathId,
  module_id: moduleId,

```

Behavior:
- If analytics POST fails, ignore and continue normal success flow (`Completed` + `router.refresh()`)

Why key uses `result.id`:
- `result.id` is the progress row ID
- Completion is one transition per row in current model, so this is stable/dedupe-safe for retries

## Acceptance Criteria

- `POST /v1/analytics/events` validates `event_name` against an allowlist and rejects invalid values.
- Endpoint supports optional `idempotency_key` and deduplicates replays by returning the existing event ID.
- `analytics_events` stores `idempotency_key` in a nullable unique column.
- Web emits best-effort `section_start` and `section_complete` analytics events from the player flow.
- Analytics emission failures do not block page render or completion UX.
- Existing API and web tests continue to pass; analytics e2e coverage expands to include validation + dedupe.

## Explicit Assumptions and Defaults

- `idempotency_key` is passed in the request body (snake_case), not a header, to keep the PR compact and consistent with current analytics payload style.
- `POST /v1/analytics/events` keeps the current response shape `{ id }` and default POST status (`201`) even for deduped replays, to avoid controller complexity in PR-11.
- PR-11 validates `payload_json` as an object but does not implement full per-event payload schemas yet.
- `player_exit` and `player_dropoff` are accepted by the API allowlist but are not emitted by the web in this PR.
- Analytics emission is best-effort and intentionally silent on failure in the player UX.
