## Public API Additions / Behavior Changes

## 1) New endpoint
- `POST /v1/unlocks/modules/:moduleId/evaluate`
- Required header: `x-user-id`

Response:
- Reuse existing `UnlockDecisionDto`:
```ts
interface UnlockDecisionDto {
  moduleId: string;
  isUnlocked: boolean;
  reasons: string[];
  requiresCredits: boolean;
  creditsCost: number;
}
```

Status codes:
- `200` on evaluate success (whether locked or unlocked decision)
- `400` missing/unknown `x-user-id`
- `404` module not found
- `500` malformed active rule config (server-side data issue)

## 2) Existing status endpoint behavior update
- `GET /v1/unlocks/modules/:moduleId/status` now:
  - supports `quiz_pass` rule evaluation
  - honors persisted grant in `user_unlocks` for module scope (unlocked immediately if row exists)

This is additive behavior, no payload shape change.

## Rule Evaluation Semantics (Decision Complete)

## Rule fetch and ordering
For both `status` and `evaluate`:
- active module rules only:
  - `scopeType = module`
  - `scopeId = moduleId`
  - `isActive = true`
- deterministic order:
  - `priority DESC`, `createdAt ASC`, `id ASC`
- AND semantics across all active rules.

## Supported rule types in PR-21
1. `prereq_sections` (existing)
2. `quiz_pass` (new)

Other types (`credits`, `min_level`) remain unmet with explicit reason:
- `Unsupported unlock rule in PR-21: <ruleType>`

## `prereq_sections` config contract (unchanged)
```json
{ "section_ids": ["<sectionId1>", "<sectionId2>"] }
```
- must be non-empty array of non-empty strings
- malformed => `500`
- unmet reason per missing completion:
  - `Complete prerequisite section: <sectionId>`

## `quiz_pass` config contract (new, locked)
```json
{ "section_ids": ["<sectionId1>", "<sectionId2>"] }
```
- must be non-empty array of non-empty strings
- malformed => `500`

Evaluation per section:
- get latest quiz attempt for `(userId, sectionId)` by:
  - `submittedAt DESC`, `attemptNo DESC`, `id ASC`
- pass condition: latest attempt exists and `passed === true`
- unmet reason when missing/failing latest:
  - `Pass quiz for section: <sectionId>`

## Unlock Persistence Semantics (Decision Complete)

## Persisted grant check
Before rule evaluation:
- check `user_unlocks` for unique `(userId, scopeType=module, scopeId=moduleId)`.
- if exists:
  - return unlocked decision immediately (`isUnlocked: true`, `reasons: []`).
  - no re-evaluation required.

## Evaluate endpoint write behavior
`POST /evaluate` flow:
1. Validate user + module.
2. If user already has module unlock row:
   - return unlocked decision (idempotent no-op).
3. Evaluate active rules.
4. If evaluation fails:
   - return locked decision, no DB writes.
5. If evaluation passes:
   - persist via `upsert` on unique key `(userId, scopeType, scopeId)`:
     - create: `reason: 'rules_satisfied'`, `unlockedAt: now()`
     - update: preserve existing row/no meaningful mutation
   - return unlocked decision.

Idempotency:
- repeated successful evaluate calls produce one row only.

## Status endpoint write behavior
- `GET /status` remains read-only (no writes).

## Explicit Assumptions and Defaults
- `quiz_pass` semantics in PR-21 are based on the **latest** attempt per section (`submittedAt`, `attemptNo` ordering), not “any historical pass”.
- Persisted module unlock in `user_unlocks` is treated as authoritative grant for status/evaluate checks.
