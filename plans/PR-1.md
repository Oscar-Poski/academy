### Prisma schema (full)

```prisma
generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

enum SectionVersionStatus {
  draft
  published
  archived
}

enum LessonBlockType {
  markdown
  callout
  code
  quiz
  checklist
}

model User {
  id        String   @id @default(cuid())
  email     String   @unique
  name      String
  createdAt DateTime @default(now()) @map("created_at")
  updatedAt DateTime @updatedAt @map("updated_at")

  @@map("users")
}

model Path {
  id          String   @id @default(cuid())
  slug        String   @unique
  title       String
  description String?
  status      String   @default("draft")
  sortOrder   Int      @default(0) @map("sort_order")
  createdAt   DateTime @default(now()) @map("created_at")
  updatedAt   DateTime @updatedAt @map("updated_at")

  modules Module[]

  @@map("paths")
}

model Module {
  id          String   @id @default(cuid())
  pathId      String   @map("path_id")
  slug        String   @unique
  title       String
  description String?
  sortOrder   Int      @default(0) @map("sort_order")
  status      String   @default("draft")
  creditsCost Int      @default(0) @map("credits_cost")
  createdAt   DateTime @default(now()) @map("created_at")
  updatedAt   DateTime @updatedAt @map("updated_at")

  path     Path      @relation(fields: [pathId], references: [id], onDelete: Cascade)
  sections Section[]

  @@index([pathId])
  @@map("modules")
}

model Section {
  id        String   @id @default(cuid())
  moduleId  String   @map("module_id")
  slug      String   @unique
  title     String
  sortOrder Int      @default(0) @map("sort_order")
  hasQuiz   Boolean  @default(false) @map("has_quiz")
  createdAt DateTime @default(now()) @map("created_at")
  updatedAt DateTime @updatedAt @map("updated_at")

  module          Module           @relation(fields: [moduleId], references: [id], onDelete: Cascade)
  sectionVersions SectionVersion[]

  @@index([moduleId])
  @@map("sections")
}

model SectionVersion {
  id            String               @id @default(cuid())
  sectionId     String               @map("section_id")
  versionNumber Int                  @map("version_number")
  status        SectionVersionStatus @default(draft)
  changeLog     String?              @map("change_log")
  publishedAt   DateTime?            @map("published_at")
  createdBy     String?              @map("created_by")
  createdAt     DateTime             @default(now()) @map("created_at")

  section      Section       @relation(fields: [sectionId], references: [id], onDelete: Cascade)
  lessonBlocks LessonBlock[]

  @@unique([sectionId, versionNumber])
  @@index([sectionId, status])
  @@map("section_versions")
}

model LessonBlock {
  id               String         @id @default(cuid())
  sectionVersionId String         @map("section_version_id")
  blockOrder       Int            @map("block_order")
  blockType        LessonBlockType @map("block_type")
  contentJson      Json           @map("content_json")
  estimatedSeconds Int?           @map("estimated_seconds")

  sectionVersion SectionVersion @relation(fields: [sectionVersionId], references: [id], onDelete: Cascade)

  @@unique([sectionVersionId, blockOrder])
  @@index([sectionVersionId])
  @@map("lesson_blocks")
}
```

### Uniqueness and published-version rule decisions

1. Slug uniqueness choice:
- `paths.slug`, `modules.slug`, `sections.slug` will each be globally unique in their own table (`@unique`).
- Justification: this simplifies routing and lookups in MVP and avoids slug collisions during early content operations.

2. Only one published section version per section:
- Enforce in DB with partial unique index in migration SQL:
  - `UNIQUE (section_id) WHERE status = 'published'`
- Justification: this is stronger than seed-only logic and prevents accidental multi-publish writes from any code path.

### Migration approach (deterministic commands)

1. Add Prisma packages in `apps/api`:
- dependency: `@prisma/client`
- dev dependency: `prisma`

2. Generate migration:
- `pnpm --filter @academy/api prisma migrate dev --name init`

3. Amend generated SQL migration to include partial unique index:
- `CREATE UNIQUE INDEX "section_versions_one_published_per_section" ON "section_versions" ("section_id") WHERE "status" = 'published';`

4. Regenerate Prisma client (if needed):
- `pnpm --filter @academy/api prisma generate`

### Seed approach

1. Strategy:
- Idempotent upserts keyed by stable identifiers (`email`, `slug`, and `(sectionId, versionNumber)`).
- For lesson blocks in published versions: deterministic replace per seeded version (`deleteMany` for that version then recreate fixed ordered blocks) to avoid duplication drift.

2. Seed contents:
- 1 user (`student@academy.local`)
- 1 path
- 1 module
- 2 sections
- each section has 2 versions (v1 published, v2 draft)
- each published version has >=3 blocks with different block types

3. Seed execution:
- `pnpm --filter @academy/api db:seed`

### Test approach

1. Keep Jest + Supertest + Nest e2e style.
2. Add test env bootstrap file:
- if `DATABASE_URL_TEST` exists, set `process.env.DATABASE_URL = process.env.DATABASE_URL_TEST`.
3. `/health` test will:
- initialize Nest app
- call `GET /health` with Supertest
- assert `{ status: 'ok', db: 'ok' }` and HTTP 200
4. Test DB expectation:
- Prefer Postgres test DB via `DATABASE_URL_TEST`
- If absent, test falls back to `DATABASE_URL`
