# PR-13 Plan: `packages/content-importer` DB Draft Upsert CLI (`--apply`) for `section_versions` + `lesson_blocks`

## Summary

PR-13 extends the new `/Users/poski/academy/packages/content-importer` package from PR-12 to support **database draft upserts** using Prisma, while preserving published/archived content.

Goals:
- Reuse PR-12 parser output (`parseContentBundle(...)`)
- Add `--apply` mode to the importer CLI
- Upsert `paths`, `modules`, `sections`, and **draft** `section_versions` + `lesson_blocks`
- Keep published versions untouched
- Make repeated runs idempotent (same bundle applied twice produces stable DB state)

Out of scope:
- No HTTP API endpoints (admin import endpoint is PR-14)
- No publish/archive workflow (PR-15)
- No content migration for in-progress learners
- No rich block parsing beyond PR-12’s single markdown block

## DB Write Behavior (Decision Complete)

### Entry flow
`--apply` mode:
1. Parse bundle via existing `parseContentBundle(rootPath)`
2. If `parseReport.errorCount > 0`:
   - do **not** write anything
   - return apply report with:
     - `applied: false`
     - `abortedReason: 'parse_errors'`
     - zeroed counts
3. If no parse errors:
   - connect Prisma
   - upsert entities in dependency order
   - return apply report with `applied: true`

### Dependency order (locked)
1. `paths`
2. `modules` (requires resolved path IDs by slug)
3. `sections` (requires resolved module IDs by slug)
4. `section_versions` + `lesson_blocks` (requires resolved section IDs by slug)

### Upsert rules by entity

#### Paths (`slug` unique)
- `upsert` by `slug`
- On create:
  - `slug`, `title`, `description`, `sortOrder`
  - `status: 'draft'` (explicit)
- On update:
  - update only `title`, `description`, `sortOrder`
  - **do not change** `status`

Rationale:
- Prevent importer from downgrading already published paths to draft

#### Modules (`slug` unique)
- `upsert` by `slug`
- On create:
  - `pathId` (resolved from `pathSlug`)
  - `slug`, `title`, `description`, `sortOrder`
  - `status: 'draft'`
- On update:
  - update `pathId`, `title`, `description`, `sortOrder`
  - **do not change** `status`
  - **do not change** `creditsCost`

Rationale:
- Content hierarchy can move if frontmatter changes path slug
- Keep unrelated unlock/economy fields stable

#### Sections (`slug` unique)
- `upsert` by `slug`
- On create:
  - `moduleId`, `slug`, `title`, `sortOrder`, `hasQuiz`
- On update:
  - update `moduleId`, `title`, `sortOrder`, `hasQuiz`

#### Section Versions (`sectionId + versionNumber` unique)
For each parsed `sectionVersion`:
1. Resolve `sectionId` from `sectionSlug`
2. Lookup existing version by composite key
3. If not found:
   - create with:
     - `sectionId`
     - `versionNumber`
     - `status: draft`
     - `changeLog`
     - `createdBy`
     - `publishedAt: null`
   - replace blocks (create initial blocks)
   - increment `sectionVersionsCreated`
4. If found and `status === draft`:
   - update:
     - `changeLog`
     - `createdBy`
     - keep `status = draft`
     - keep `publishedAt` unchanged/null (set `publishedAt: null` is acceptable)
   - replace all `lesson_blocks` for that version
   - increment `sectionVersionsUpdated`
5. If found and `status !== draft` (`published` or `archived`):
   - **skip** this version write entirely
   - do not modify blocks
   - record skip in `skips[]`
   - increment `sectionVersionsSkippedNonDraft`

### Lesson Block replacement pattern (for created/updated draft versions)
Use a transaction per section version:
- `deleteMany({ where: { sectionVersionId } })`
- `createMany({ data: parsed blocks mapped to DB columns })`

Counting:
- `lessonBlocksReplaced += parsedBlocks.length` for each created/updated draft version

### Atomicity policy (locked)
- Entity-level writes are **not** wrapped in one global transaction for the entire bundle
- Each section-version block replacement is transactional
- Path/module/section upserts run sequentially before version writes

Reason:
- Simpler failure handling/debugging for PR-13
- Idempotent reruns recover safely
- PR-14 admin endpoint can add job-level orchestration later

## Skip / Error Semantics

### Parse errors (PR-12 validation)
- Abort all DB writes in `--apply`
- Return `applied: false`, `abortedReason: 'parse_errors'`

### Write-time missing dependencies
Should not happen if parse report is internally consistent, but if encountered:
- Throw and fail CLI with non-zero exit
- Include clear error (e.g., `Resolved path slug missing during apply: <slug>`)

### Non-draft section version collision
- Not a hard failure
- Recorded in `skips[]`
- Continue processing remaining section versions

Reason:
- Preserves published/archived versions
- Allows bundle apply to partially succeed for new/draft versions

- `1` usage error / unknown args

- `0` dry-run with warnings only
- `0` apply success (including non-draft skips)

## Explicit Assumptions and Defaults

- PR-13 continues using PR-12’s bundle contract: one `.md/.mdx` file == one section-version draft document.
- The importer writes to the DB referenced by `DATABASE_URL` in the current shell environment.
- Parse errors abort all writes in `--apply` mode.
- Warnings do not block writes.
- Existing `paths/modules` statuses are preserved on update; importer only sets `status: 'draft'` on create.
- Existing non-draft section versions (`published`, `archived`) are skipped, not overwritten or downgraded.
- Lesson block replacement for draft versions uses `deleteMany + createMany` within a transaction, mirroring the seed script pattern.
