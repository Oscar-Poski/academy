# PR-15 Plan: Admin Section Version Listing/Preview + Publish Endpoint

## Summary

PR-15 adds admin read + publish operations for section versions in `apps/api`:

- `GET /v1/admin/sections/:sectionId/versions`
- `GET /v1/admin/sections/:sectionId/versions/:versionId`
- `POST /v1/admin/sections/:sectionId/publish/:versionId`

Goals:
- Let admins inspect draft/published/archived versions for a section.
- Preview a specific version’s blocks before publishing.
- Publish a draft version while archiving the previously published version(s) in a transaction.
- Preserve version immutability/pinning semantics (pinned users stay on archived version; new users get newly published version).

Out of scope:
- Auth/role checks
- Admin UI
- Version migration of in-progress learners
- Content editing/import changes (already PR-12/13/14)
- Additional schema changes

## Current State (Repo-Grounded)

- `AdminModule` already exists with `POST /v1/admin/content/import`.
- `ContentService.getSection(...)` resolves version by:
  - pinned `user_section_progress.sectionVersionId` if version is `published` or `archived`
  - otherwise latest `published`
- This means PR-15 publish flow must archive the old published version rather than modifying/deleting it.
- `section_versions` schema already supports:
  - `status` enum (`draft`, `published`, `archived`)
  - `publishedAt`
  - immutable version identity via `(sectionId, versionNumber)`

### `GET /v1/admin/sections/:sectionId/versions` Response
Return an array of version summaries (ordered by `versionNumber DESC`):

```ts
interface SectionVersionSummaryDto {
  id: string;
  sectionId: string;
  versionNumber: number;
  status: 'draft' | 'published' | 'archived';
  changeLog: string | null;
  createdBy: string | null;
  createdAt: string;      // Date serialized by Nest
  publishedAt: string | null;
  blockCount: number;
}
```

### `GET /v1/admin/sections/:sectionId/versions/:versionId` Response
Return version metadata + blocks (preview payload):

```ts
interface SectionVersionDetailDto {
  id: string;
  sectionId: string;
  versionNumber: number;
  status: 'draft' | 'published' | 'archived';
  changeLog: string | null;
  createdBy: string | null;
  createdAt: string;
  publishedAt: string | null;
  lessonBlocks: Array<{
    id: string;
    blockOrder: number;
    blockType: 'markdown' | 'callout' | 'code' | 'quiz' | 'checklist';
    contentJson: unknown;
    estimatedSeconds: number | null;
  }>;
}
```

### `POST /v1/admin/sections/:sectionId/publish/:versionId` Response
Return a compact publish result report:

```ts
interface PublishSectionVersionResponseDto {
  sectionId: string;
  versionId: string;
  versionNumber: number;
  status: 'published';
  publishedAt: string;
  archivedVersionIds: string[];
}
```

### HTTP Status Codes (Locked)
- `GET` endpoints: `200`, `404`
- `POST publish`: `200` on success (explicit `@HttpCode(200)`)
- `404` if section/version missing or version does not belong to section
- `409` if target version is not `draft`
- `500` for unexpected failures

## Service Plan (`admin.service.ts`)

Add methods:

- `listSectionVersions(sectionId: string): Promise<SectionVersionSummaryDto[]>`
- `getSectionVersion(sectionId: string, versionId: string): Promise<SectionVersionDetailDto>`
- `publishSectionVersion(sectionId: string, versionId: string): Promise<PublishSectionVersionResponseDto>`

### Shared helper methods (recommended)
- `assertSectionExists(sectionId: string): Promise<void>`
- `getSectionVersionOrThrow(sectionId: string, versionId: string)` (ensures ownership)
- `toSectionVersionSummaryDto(...)`
- `toSectionVersionDetailDto(...)`

### `listSectionVersions` behavior (locked)
1. Verify section exists (`404` if not).
2. Query all versions for section with `_count` of lesson blocks.
3. Order by:
   - `versionNumber DESC`
   - fallback `createdAt DESC` (optional)
4. Return summaries.

### `getSectionVersion` behavior (locked)
1. Query version by `id` + `sectionId`.
2. `404` if not found.
3. Include ordered `lessonBlocks`:
   - `blockOrder ASC`, `id ASC`
4. Return detail DTO.

### `publishSectionVersion` behavior (locked)
1. Fetch target version by `id` + `sectionId`.
2. `404` if missing.
3. If target status is not `draft`, throw `ConflictException` (`409`).
4. In a single Prisma transaction:
   - find currently published versions for same section excluding target (`status = published`)
   - archive all of them (`status = archived`)
   - publish target version (`status = published`, `publishedAt = now`)
5. Return publish result with:
   - target version metadata after update
   - list of archived version IDs

### Publish semantics (important decisions)
- Archive **all** currently published versions for that section (not just one) to self-heal data anomalies.
- Preserve archived versions and their blocks.
- Preserve archived versions’ existing `publishedAt` timestamps (do not null them).
- Target must be `draft`; publishing a `published`/`archived` version returns `409`.

## Data Consistency / Versioning Semantics

### Why archiving preserves pinning
`ContentService.getSection(...)` pinned-version path accepts statuses `[published, archived]`, so:
- users pinned to the previously published version continue seeing it after publish (now archived)
- public/no-progress users resolve to the new published version

### Transaction scope
Use a single Prisma `$transaction` in `publishSectionVersion` for:
- archiving old published versions
- publishing target version

This prevents transient “no published version” / “two published versions” windows within one request.

  - correct version ID/section ID

  - includes `contentJson`, `estimatedSeconds`

#### 3. `POST publish` archives previous published and publishes target draft
- Start with `v1 = published`, `v2 = draft`
- Publish `v2`
- Expect `200`
- Assert DB:

  - `v2.publishedAt` set

- `GET /v1/sections/:sectionId` with `x-user-id` pinned user -> still returns `v1`
- `GET /v1/sections/:sectionId` public -> returns `v2`
- (This is the key regression guard for PR-15)

#### 5. `POST publish` rejects non-draft target with `409`
- Try publishing `v1` when already `published` (or archived)
- Expect `409`

#### 6. `GET` / `POST` return `404` for invalid section/version combinations
- nonexistent section ID
- valid section + version ID from another section
- Expect `404`

## Acceptance Criteria

- `GET /v1/admin/sections/:sectionId/versions` returns version summaries with block counts.
- `GET /v1/admin/sections/:sectionId/versions/:versionId` returns version preview with lesson blocks.
- `POST /v1/admin/sections/:sectionId/publish/:versionId` publishes a draft and archives prior published version(s) in one transaction.
- Publishing preserves pinned-user behavior (old version archived, still resolvable via pinned progress).
- Non-draft publish attempts return `409`.
- Invalid section/version IDs or mismatched ownership return `404`.
- No Prisma schema/migration changes.
- Existing content/progress/versioning behavior remains backward compatible.

- Empty-block draft versions are allowed to be published in PR-15 (no content completeness validation yet).
