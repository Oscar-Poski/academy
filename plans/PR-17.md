## Public API Additions

## New endpoint
- `POST /v1/quizzes/sections/:sectionId/attempts`

### Headers
- Required: `x-user-id` (existing temporary strategy)

### Request body
```ts
interface QuizSubmissionDto {
  answers: Array<{
    question_id: string;
    selected_option: string;
  }>;
}
```

### Response body
```ts
interface QuizAttemptResultDto {
  attemptId: string;
  userId: string;
  sectionId: string;
  sectionVersionId: string;
  attemptNo: number;
  score: number;
  maxScore: number;
  passed: boolean;
  submittedAt: string;
  feedback: Array<{
    questionId: string;
    isCorrect: boolean;
    awardedPoints: number;
    expectedOption: string;
    selectedOption: string | null;
    explanation: string | null;
  }>;
}
```

## Scoring Rules (locked for PR-17)

1. **Only MCQ questions are graded**.
- Source set = `questions` where:
  - `sectionVersionId` = resolved quiz version
  - `type = mcq`
- `short_answer` questions are ignored in PR-17 (not part of `maxScore` and not included in feedback).

2. **Section version resolution**
- If user has `user_section_progress` for the section, use `sectionVersionId` from progress when that version status is `published` or `archived`.
- Otherwise use latest `published` version for the section.
- Same behavior family as content version resolution.

3. **Answer matching**
- Per MCQ question, expected option from `answerKeyJson.correct_option` (string).
- Submitted option = `selected_option` from request.
- Correct if exact string match after trim (case-sensitive).

4. **Unanswered MCQ**
- Treated as incorrect.
- Included in feedback with `selectedOption: null` and `awardedPoints: 0`.

5. **Score/pass**
- `score = sum(awardedPoints)`
- `maxScore = sum(points)`
- `passed = score / maxScore >= 0.7` (fixed threshold for PR-17)
- If `maxScore === 0`, reject with `400` (`No MCQ questions available for this section version`).

---

## Validation and Error Behavior

### Request validation
- `answers` must be an array.
- Each answer item must include non-empty `question_id` and non-empty `selected_option`.
- Duplicate `question_id` in payload => `400`.

### Domain validation
- Unknown/missing `x-user-id` => `400`.
- Section not found => `404`.
- Resolved section version missing => `404`.
- If submission includes `question_id` not in the resolved MCQ question set => `400`.
- If question config is invalid (`answerKeyJson.correct_option` missing/non-string) => `500` (misconfigured server data).

### Persistence behavior
- Always creates a new attempt row on valid request.
- `attemptNo` increments per `(userId, sectionId)` as `max(existing attemptNo) + 1`.
- Saves one `quiz_attempt_answers` row per MCQ question (including unanswered ones with empty/null selected value representation in `answer_json`).

### Required test cases

1. **Happy path (all correct MCQ answers)**
- Submit correct answers for seeded `request-response-cycle` MCQ questions.
- Expect `201`, `passed=true`, `score=maxScore`.
- Assert DB:
  - one `quiz_attempts` row created
  - `quiz_attempt_answers` count equals MCQ question count
  - `attemptNo=1` for first attempt

2. **Incorrect/mixed answers**
- Submit one correct, one incorrect.
- Expect `201`, lower score, pass/fail according to 70% threshold.
- Assert persisted awarded points reflect correctness.

3. **Unanswered MCQ handling**
- Omit one MCQ from payload.
- Expect attempt still succeeds.
- Missing question appears in feedback as incorrect with `selectedOption: null`.

4. **Duplicate `question_id` in payload**
- Expect `400`.

5. **Unknown question_id in payload**
- Expect `400`.

6. **Attempt number increments**
- Submit twice for same user+section.
- Assert second response `attemptNo=2` and second attempt row exists.

7. **Missing or unknown user header**
- Missing `x-user-id` => `400`
- Unknown user => `400`

Optional low-friction:
8. **No MCQ questions in resolved version**
- create temporary section version with only `short_answer` and submit
- expect `400` no MCQ available
